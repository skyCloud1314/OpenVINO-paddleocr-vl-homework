# 【PaddlePaddle Hackathon 10 方案说明】PaperMind — 基于 OpenVINO 的学术论文智能研读平台

> 赛道：Intel · 基于 OpenVINO 的多模态文档理解与智能应用开发

---

## 1. 项目概述

### 1.1 一句话描述

**PaperMind** 是一款面向科研人员的学术论文智能研读平台——上传 PDF 论文后，通过 OpenVINO 加速的 PaddleOCR-VL 将每页转化为高保真结构化 Markdown（含 LaTeX 公式、表格、图注），再对接下游 LLM 完成**论文摘要、智能问答、公式解读、表格分析、全文翻译、一键导出**六大功能。全流程支持 **CPU / iGPU / NPU 多设备实时切换**，内置**性能对比面板**，同时提供 **OpenVINO Model Server (OVMS) 远程推理**方案。

### 1.2 与现有项目的差异化

| 维度 | 现有 paddleocr_vl Notebook | 现有 demos (Virtual AI Assistant 等) | **PaperMind (本方案)** |
|------|---------------------------|--------------------------------------|----------------------|
| 定位 | 模型转换 + 单图推理演示 | 通用对话 / 图像描述 / 物体检测 | **学术论文全流程深度解析 + 智能研读** |
| 输入 | 单张图片 | 文本 / 摄像头 | **多页 PDF 论文文档** |
| 输出 | 原始 OCR 文本 | 对话文本 / 检测框 | **结构化 Markdown + LaTeX + 六大下游智能任务** |
| 下游任务 | 无 | RAG 对话 (通用) | **摘要 / 问答 / 公式解读 / 表格分析 / 翻译 / 导出** |
| OCR 后处理 | 无 | 无 | **多栏排版重排 + 公式识别 + 表格结构化** |
| 部署形态 | Jupyter Notebook | 单机 Gradio | **单机 Gradio + OVMS 远程推理** |
| 设备支持 | 单设备 | 单设备 | **CPU / iGPU / NPU 动态切换 + 性能对比** |

**结论：本方案在应用场景、输入模态、后处理逻辑、下游任务链、部署形态和设备支持上与所有现有内容完全不重复。**

## 2. 问题与动机

### 2.1 科研阅读的现实痛点

科研人员平均每周需要阅读 5–15 篇论文，但现有工具在处理学术 PDF 时问题严重：

| 痛点 | 具体表现 | 影响 |
|------|---------|------|
| **公式乱码** | 传统 OCR（Tesseract、PyMuPDF text extraction）无法识别行内/行间数学公式 | LLM 无法理解论文核心算法推导 |
| **排版错位** | 双栏 PDF 被逐行提取，左右两栏文本交错 | 摘要逻辑混乱、不可用 |
| **表格语义丢失** | 复杂表格（合并单元格、跨页表格）被提取为纯文本 | 无法进行实验数据对比分析 |
| **图文断裂** | 图片标题（Caption）与正文失去关联 | LLM 无法引用图表进行推理 |
| **语言壁垒** | 非英语母语科研人员需反复切换翻译工具 | 阅读效率低下 |
| **隐私风险** | 使用云端 API（如 GPT-4V）处理未发表论文 | 科研成果可能泄露 |

### 2.2 为什么选择 PaddleOCR-VL + OpenVINO

- **PaddleOCR-VL**：多模态视觉语言模型，不仅"看到"文字，更"理解"文档布局——直接从像素级输入生成带结构标记的文本，天然解决公式识别和表格理解问题
- **OpenVINO**：通过 NNCF 量化（INT4/INT8）和 Stateful 模型优化，使 2B+ 参数多模态模型在个人 PC（Intel 酷睿 Ultra）上高效运行，实现完全本地化的隐私保护
- **OVMS**：通过 OpenVINO Model Server 提供 gRPC/REST API 服务化推理，支持实验室团队共享和远程访问

## 4. 六大核心功能

### 4.1 高保真论文解析

| 能力 | 实现方式 |
|------|---------|
| 数学公式 → LaTeX | PaddleOCR-VL 直接生成 `$...$` / `$$...$$` |
| 复杂表格 → Markdown | 保留合并单元格语义，自动补全分隔行 |
| 图片标题 → 关联保留 | `**Figure X:** ...` 格式与正文绑定 |
| 双栏排版 → 正确顺序 | 空白区域分析 + 分栏裁剪 + 按序拼接 |

### 4.2 论文摘要生成

输入完整 Markdown 文档，自动生成四维结构化摘要：
- **研究目标**：论文要解决什么问题
- **核心方法**：技术路线和关键算法
- **实验结果**：定量指标和对比数据
- **主要结论**：核心贡献和创新点

### 4.3 智能问答

- 基于论文全文上下文精准问答，引用具体章节/公式/表格
- 自动截断上下文以适配 LLM Token 限制
- 拒绝回答论文中不存在的内容，避免幻觉

### 4.4 公式解读

- 用户粘贴 LaTeX 公式，LLM 结合论文上下文逐步解释
- 解析每个符号的含义和推导过程
- 说明公式在论文中的角色

### 4.5 表格数据分析

- 解析 Markdown 表格结构，提取行列关系
- LLM 进行趋势分析（如"方法 A 在指标 X 上比方法 B 高 3.2%"）
- 自动识别最优结果并给出结论

### 4.6 全文翻译 (中↔英)

- **保结构翻译**：保持 Markdown 格式、LaTeX 公式、表格结构不变
- 仅翻译自然语言文本，术语保留原文标注
- 支持段落级增量翻译，翻译过程实时显示

### 4.7 一键导出

- **PDF**：Markdown 渲染为排版精良的 PDF
- **DOCX**：python-docx 生成 Word 文档，保留标题层级和表格
- **HTML**：带 MathJax 渲染的完整 HTML 页面
- 可选择导出内容：原文 / 翻译 / 摘要 / 全部

## 5. OpenVINO 优化策略（核心亮点）

### 5.1 模型量化

| 子模型 | 原始精度 | 优化策略 | 预期效果 |
|--------|---------|---------|---------|
| Vision Encoder | FP32 | NNCF INT8 PTQ (smooth_quant_alpha=0.6) | 推理速度 ~2x，精度损失 < 1% |
| Vision MLP | FP32 | FP16 推理 | 内存减半 |
| LLM Embedding | FP32 | FP16 推理 | 内存减半 |
| LLM Decoder | FP32 | NNCF INT4 AWQ (group_size=64, ratio=0.9) | 内存 ~4x↓，适配 16GB 设备 |

### 5.2 Stateful 模型优化

利用 `patch_stateful()` 将 KV-Cache 内化为模型内部状态：
- 避免每次推理重复传递 `past_key_values` 张量
- 在长文档逐页生成场景下显著降低内存带宽压力
- 支持 Beam Search 和 Greedy Decoding 两种策略

### 5.3 多设备调度（加分项）

```
┌──────────┬──────────────────────┬─────────────────────────┐
│ 设备      │ 负载                 │ 优势                    │
├──────────┼──────────────────────┼─────────────────────────┤
│ CPU      │ LLM Decoder (INT4)   │ AVX-512/AMX 矩阵加速   │
│ iGPU     │ Vision Encoder (INT8)│ EU 并行图像特征提取     │
│ NPU      │ Vision Encoder (备选)│ 极低功耗，持续运行      │
└──────────┴──────────────────────┴─────────────────────────┘
```

**实现方式**：Gradio 下拉菜单选择设备，运行时通过 `ov.Core().compile_model(model, device)` 动态切换。系统自动检测可用设备列表，灰化不可用选项。

### 5.4 性能基准测试（加分项）

内置 Benchmark 面板，支持以下对比维度：

| 对比维度 | 测试项目 |
|---------|---------|
| 精度对比 | FP32 vs INT8 vs INT4 延迟/精度 |
| 设备对比 | CPU vs iGPU vs NPU 延迟/吞吐 |
| 端到端对比 | 单页延迟、多页总延迟、LLM 生成吞吐 |
| 内存对比 | 不同精度/设备组合的峰值内存 |

每次推理自动更新性能指标，在 Gradio 底部面板实时展示。

### 5.5 OpenVINO Model Server 部署（加分项）

```
     本地模式                          OVMS 模式
┌──────────────┐              ┌──────────────────────┐
│  Gradio App  │              │     Gradio App       │
│      ↓       │              │         ↓            │
│  OV Runtime  │              │   gRPC Client        │
│  (本地推理)   │              │         ↓            │
└──────────────┘              │   OVMS Server        │
                              │   PaddleOCR-VL IR    │
                              │   gRPC:9000          │
                              └──────────────────────┘
```

提供完整的 OVMS 部署方案：
- OVMS 服务端模型配置文件（`model_config.json`），包含模型路径、精度、批大小
- gRPC 客户端封装（`ovms_client.py`），与本地推理接口完全兼容，应用层无需修改
- 详细的 OVMS 安装与启动指南（README 中说明 pip 安装 / 手动部署方式）
- 支持场景：实验室共享服务器 → 多人远程调用同一模型服务

> 注：OVMS 的部署方式不依赖 Docker。可通过 `pip install ovmsclient` 安装客户端，服务端可直接在目标机器上以原生进程方式运行，也可通过预编译二进制包部署。

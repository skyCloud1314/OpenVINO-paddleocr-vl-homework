# PaperMind：基于 OpenVINO 与 PaddleOCR-VL 的科研论文智能研读助手

> **PaddlePaddle Hackathon 10 · Intel OpenVINO 赛道 · RFC 方案说明**

---

## 1. 项目概述

### 1.1 一句话描述

PaperMind 是一款面向科研人员的**本地化论文深度研读工具**——用户上传学术 PDF，系统通过 OpenVINO 加速的 PaddleOCR-VL 模型将每页论文转化为高保真结构化 Markdown（含 LaTeX 公式、表格、图注），并对接下游 LLM 实现**论文摘要、公式解读、表格分析、跨章节智能问答**四大核心功能。

### 1.2 与现有项目的差异化

| 维度 | 现有 paddleocr_vl Notebook | 现有 demos (Virtual AI Assistant 等) | **PaperMind (本方案)** |
|------|---------------------------|--------------------------------------|----------------------|
| 定位 | 模型转换与单图推理演示 | 通用对话 / 图像描述 / 物体检测 | **学术论文全流程深度解析** |
| 输入 | 单张图片 | 文本 / 摄像头 | **多页 PDF 文档** |
| 输出 | 原始 OCR 文本 | 对话文本 / 检测框 | **结构化 Markdown + LaTeX + 智能问答** |
| 下游任务 | 无 | RAG 对话 (通用) | **论文摘要 / 公式推导解读 / 表格数据分析 / 跨章节 Q&A** |
| OCR 后处理 | 无 | 无 | **多栏排版重排 + 公式区域识别 + 表格结构化** |

**结论：本方案在应用场景、输入模态、后处理逻辑和下游任务链上与所有现有内容完全不重复。**

## 2. 问题与动机

### 2.1 科研阅读的现实痛点

科研人员平均每周需要阅读 5-15 篇论文，但现有工具在处理学术 PDF 时面临以下严重问题：

| 痛点 | 具体表现 | 影响 |
|------|---------|------|
| **公式乱码** | 传统 OCR（如 Tesseract、PyMuPDF）无法识别行内/行间数学公式 | LLM 无法理解论文的核心算法推导 |
| **排版错位** | 双栏 PDF 被逐行提取，左右两栏文本交错 | 生成的摘要逻辑混乱、不可用 |
| **表格语义丢失** | 复杂表格（合并单元格、跨页表格）被提取为纯文本 | 无法进行数据对比和趋势分析 |
| **图文断裂** | 图片标题（Caption）与正文失去关联 | LLM 无法引用图表进行推理 |
| **隐私风险** | 使用云端 API（如 GPT-4V）处理未发表论文 | 科研成果存在泄露风险 |

### 2.2 为什么 PaddleOCR-VL + OpenVINO 是最佳方案

- **PaddleOCR-VL**：作为多模态视觉语言模型，它不仅"看到"文字，更"理解"文档布局。它能直接从像素级输入生成带结构标记的文本，天然解决公式识别和表格理解问题。
- **OpenVINO**：通过 NNCF 量化（INT4/INT8）和 Stateful 模型优化，使这一 2B+ 参数的多模态模型能够在个人 PC（Intel 酷睿 Ultra 处理器）上高效运行，实现完全本地化的隐私保护。

### 3.2 核心模块设计

#### 模块 A：PDF 预处理器 (`pdf_preprocessor.py`)

- 使用 `PyMuPDF (fitz)` 将 PDF 逐页渲染为高分辨率图片（300 DPI）。
- 自动裁剪页眉/页脚/页码等干扰区域。
- 对双栏论文进行分栏检测（基于空白区域分析），确保左栏 → 右栏的正确阅读顺序。

#### 模块 B：PaddleOCR-VL OpenVINO 推理引擎 (`ov_inference.py`)

- 基于 `ov_paddleocr_vl.py` 中的 `OVPaddleOCRVLForCausalLM` 类进行封装。
- 针对每页图片构造专用 Prompt：
  ```
  请将这张学术论文页面的内容转换为结构化 Markdown 格式。
  要求：
  1. 数学公式使用 LaTeX 语法（行内用 $...$，行间用 $$...$$）
  2. 表格使用 Markdown 表格语法
  3. 图片标题以 "**Figure X:** ..." 格式保留
  4. 保持原文的章节层级结构（#, ##, ###）
  ```
- 通过 `chat()` 接口调用 OpenVINO Stateful 模型完成推理。

#### 模块 C：结构化后处理器 (`post_processor.py`)

- **LaTeX 校验**：检查生成的 LaTeX 公式是否可正确渲染，修复常见错误（如未闭合的花括号）。
- **表格格式化**：将 OCR 输出的表格规范化为标准 Markdown 表格，对齐列宽。
- **章节合并**：将多页解析结果按章节编号 (Section 1, 2, 3...) 拼接为完整的论文 Markdown 文档。

#### 模块 D：下游 LLM Agent (`paper_agent.py`)

- **摘要生成**：基于完整 Markdown 文档生成结构化摘要（包含研究目标、方法、实验结果、结论）。
- **公式解读**：选中任意公式，LLM 逐步解释每个符号的含义和推导过程。
- **表格分析**：将 Markdown 表格转为结构化数据，LLM 进行趋势对比分析。
- **智能问答**：基于 RAG（向量检索增强生成），用户可以自然语言提问论文中的任意内容。

## 4. OpenVINO 优化策略

### 4.1 模型量化方案

| 子模型 | 原始精度 | 优化策略 | 预期效果 |
|--------|---------|---------|---------|
| Vision Encoder | FP32 | NNCF INT8 Post-Training Quantization (`nncf.quantize`, `smooth_quant_alpha=0.6`) | 推理速度提升 ~2x，精度损失 < 1% |
| Vision MLP | FP32 | FP16 精度推理 | 内存减半 |
| LLM Embedding | FP32 | FP16 精度推理 | 内存减半 |
| LLM Decoder | FP32 | NNCF INT4 Asymmetric Weight Compression (`group_size=64, ratio=0.9`) | 内存降低 ~4x，适配 16GB 设备 |

### 4.2 Stateful 模型优化

利用 `ov_paddleocr_vl.py` 中的 `patch_stateful()` 函数，将 KV-Cache 内化为模型内部状态变量，避免每次推理时重复传递巨大的 past_key_values 张量。在长文档逐页生成场景下，此优化可显著降低内存带宽压力。

### 4.3 设备调度策略

```
CPU  →  LLM Decoder (INT4)   # 利用 AVX-512/AMX 加速矩阵运算
iGPU →  Vision Encoder (INT8) # 利用 GPU 并行加速图像特征提取
NPU  →  Vision Encoder (备选)  # Intel 酷睿 Ultra NPU 加速，进一步降低功耗
```

用户可通过 Gradio 界面下拉菜单自由选择设备，系统自动展示不同设备组合下的性能指标。

### 4.4 性能基准测试计划

| 指标 | 测试方法 | 目标值 |
|------|---------|-------|
| 单页解析延迟 (Latency) | 10 页论文取平均 | < 5s / 页 (CPU), < 3s / 页 (iGPU) |
| LLM 生成吞吐 (Tokens/s) | 生成 512 tokens 取平均 | > 15 tokens/s (INT4) |
| 内存峰值 (Peak RAM) | 处理 20 页论文时监测 | < 12 GB |
| 公式识别准确率 | 50 个公式人工评估 | > 85% |
| 端到端量化前后对比 | FP32 vs INT4 延迟/精度对比 | 延迟降低 > 50%, 精度损失 < 3% |

### 5.2 交互流程

1. 用户上传 PDF 论文文件。
2. 系统自动将 PDF 逐页渲染为图片，调用 PaddleOCR-VL (OpenVINO) 进行解析。
3. 左侧展示 PDF 原始页面，右侧实时展示对应的 Markdown 渲染结果。
4. 下方 Agent 面板提供四个快捷按钮和一个自由输入框。
5. 底部实时展示推理性能指标（Vision 编码耗时、LLM 生成耗时、Tokens/s）。
